# ost-network-tools 可行性建设建议

## 0. 当前仓库判断

仓库：`https://github.com/apllozhang/ost-network-tools`

当前状态判断：

- 仓库已创建。
- 默认分支为 `main`。
- 仓库当前 size 为 `0`。
- 暂未看到可用的 `README.md`、`package.json`、`src/`、`app/`、`backend/`、`docker-compose.yml` 等工程文件。
- 因此当前不适合直接进入“代码重构建议”阶段，更适合进入“产品骨架 + 技术架构 + MVP 路线”阶段。

结论：

> `ost-network-tools` 当前应按一个全新的网络运维工具平台仓库来规划，而不是按已有成熟代码库来改造。

# 1. 产品定位建议

`ost-network-tools` 不建议只做成一个网络工具集合，例如 Ping、Traceroute、端口扫描这类小工具合集。

更建议定位为：

> 面向网络运维场景的轻量级设备监控、链路诊断、告警处置和可视化仪表盘平台。

建议产品方向：

```text
网络工具箱
  -> 网络诊断工作台
    -> 设备监控平台
      -> 网络运维驾驶舱
        -> 企业级网络运维管理平台
```

第一阶段不要追求大而全，建议先做成：

```text
首页驾驶舱
+ 设备列表
+ 设备详情
+ Ping / TCP / HTTP 探测
+ 链路质量趋势
+ 告警中心
+ 日志与事件时间线
```

# 2. 推荐技术栈

## 2.1 前端技术栈

建议采用：

| 模块 | 推荐方案 | 原因 |
|---|---|---|
| 框架 | React + TypeScript | 生态成熟，适合仪表盘和管理后台 |
| 构建 | Vite | 轻量、启动快，适合独立项目 |
| 路由 | React Router 或 TanStack Router | 多页面管理后台需要清晰路由 |
| UI 组件 | Ant Design / shadcn/ui 二选一 | Ant Design 适合企业后台，shadcn/ui 更现代 |
| 图表 | Apache ECharts | 适合流量、延迟、丢包、状态趋势 |
| 表格 | TanStack Table / Ant Design Table | 设备、告警、日志都需要复杂表格 |
| 状态管理 | Zustand | 简洁，适合全局筛选、主题、用户状态 |
| 请求层 | TanStack Query | 适合指标、告警、设备列表的缓存与刷新 |
| 样式 | Tailwind CSS 或 CSS Modules | 便于统一视觉规范 |
| 实时通信 | WebSocket / SSE | 告警、设备状态、实时日志需要推送 |

## 2.2 后端技术栈

如果你想快速落地，推荐：

| 模块 | 推荐方案 | 原因 |
|---|---|---|
| 后端语言 | Python FastAPI | 网络工具、异步探测、SNMP、脚本集成方便 |
| API | REST API | 前后端分离简单直接 |
| 实时接口 | WebSocket / SSE | 推送告警和探测状态 |
| 数据库 | PostgreSQL | 设备、告警、任务、资产数据 |
| 时序数据 | VictoriaMetrics / Prometheus / TimescaleDB | 存储延迟、丢包、流量、CPU 等指标 |
| 缓存 | Redis | 任务队列、临时状态、实时数据缓存 |
| 任务队列 | Celery / Dramatiq / RQ | Ping、Traceroute、SNMP Polling、巡检任务 |
| 部署 | Docker Compose | 第一阶段最容易交付 |
| 日志 | Loki / OpenSearch 可选 | 后续扩展日志检索 |

## 2.3 最小推荐组合

第一阶段最推荐：

```text
Frontend:
  React + TypeScript + Vite + Ant Design + ECharts + TanStack Query

Backend:
  Python + FastAPI + PostgreSQL + Redis + Celery

Deploy:
  Docker Compose
```

# 4. MVP 功能建议

## 4.1 第一阶段 MVP 目标

第一阶段不要做完整 NMS，而是做：

> 可演示、可部署、可扩展的轻量网络运维驾驶舱。

MVP 必须包含：

| 模块 | 是否必须 | 说明 |
|---|---|---|
| 首页驾驶舱 | 必须 | 展示全局健康状态 |
| 设备列表 | 必须 | 展示设备状态、IP、站点、标签 |
| 设备详情 | 必须 | 展示设备基础信息、探测趋势、事件 |
| 探测任务 | 必须 | Ping、TCP Port、HTTP Check |
| 告警中心 | 必须 | 当前告警、历史告警、确认、关闭 |
| 链路质量 | 建议 | 延迟、丢包、抖动 |
| 日志 / 事件 | 建议 | 展示探测失败、恢复、配置变更 |
| 用户权限 | 后置 | 第一阶段可先做简单登录 |
| 拓扑图 | 后置 | 第二阶段再做 |
| SNMP / gNMI | 后置 | 第一阶段先预留接口 |

# 5. 页面设计建议

## P01：首页驾驶舱

### 页面目标

首页只回答一个问题：

> 当前网络是否健康，哪里需要立刻处理？

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ TopBar：搜索 / 时间范围 / 自动刷新 / 主题切换 / 用户菜单        │
├──────────────────────────────────────────────────────────────┤
│ KPI：设备总数 / 在线设备 / 离线设备 / 当前告警 / 平均延迟 / 可用率│
├───────────────────────────────┬──────────────────────────────┤
│ 左：全网健康趋势               │ 右：当前严重告警                │
├───────────────────────────────┬──────────────────────────────┤
│ 左：链路质量趋势               │ 右：慢响应设备排行              │
├───────────────────────────────┴──────────────────────────────┤
│ 底部：按站点 / 类型 / 标签分组的设备健康矩阵                   │
└──────────────────────────────────────────────────────────────┘
```

### 组件拆分

```text
DashboardPage
├─ DashboardHeader
│  ├─ TimeRangeSelector
│  ├─ AutoRefreshToggle
│  └─ GlobalSearch
├─ KpiGrid
│  ├─ TotalDevicesCard
│  ├─ OnlineDevicesCard
│  ├─ OfflineDevicesCard
│  ├─ ActiveAlertsCard
│  ├─ AvgLatencyCard
│  └─ AvailabilityCard
├─ TrendSection
│  ├─ HealthTrendChart
│  ├─ LatencyTrendChart
│  └─ PacketLossTrendChart
├─ AlertSection
│  ├─ CriticalAlertsCard
│  └─ RecentRecoveriesCard
└─ GroupHealthSection
   ├─ SiteHealthMatrix
   ├─ DeviceTypeHealthMatrix
   └─ TagHealthMatrix
```

### UI 建议

- KPI 卡片控制在 4 到 6 张。
- 不要首页堆满图表。
- 红色只用于故障，黄色用于警告，灰色用于未知。
- 所有 KPI 都必须能点击下钻。
- 首页默认时间范围建议为最近 24 小时。
- 支持自动刷新，但必须允许暂停。

## P02：设备列表页

### 页面目标

帮助用户快速找到异常设备。

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：设备监控 / 新增设备 / 导入 / 导出                  │
├──────────────────────────────────────────────────────────────┤

└──────────────────────────────────────────────────────────────┘
```

### 表格字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 状态 | StatusBadge | 正常、警告、故障、未知、维护中 |
| 设备名 | Link | 点击进入详情 |

| 类型 | Tag | Router、Switch、Firewall、Server、AP |
| 站点 | Tag | 北京、上海、IDC、分支 |

- 默认每页 50 条。
- 支持列显示/隐藏。
- 支持保存筛选条件。
- 点击设备名进入详情页。
- 点击告警数量进入告警中心并带过滤条件。

## P04：网络工具页

### 页面目标

这是 `ost-network-tools` 的特色页面，可以体现“工具属性”。

### 推荐工具

| 工具 | 第一阶段是否实现 | 说明 |
|---|---|---|
| Ping | 必须 | ICMP 连通性测试 |
| TCP Port Check | 必须 | 检测端口连通性 |
| HTTP Check | 必须 | 检测 URL 可用性和状态码 |
| DNS Lookup | 建议 | 查询 A、AAAA、CNAME、MX |
| Traceroute | 建议 | 路径追踪 |
| Whois | 后置 | 域名和 IP 信息 |
| SSL Certificate Check | 后置 | 证书过期检查 |
| SNMP Walk | 后置 | 网络设备指标采集 |
| Subnet Calculator | 后置 | 网段计算 |
| IP Conflict Check | 后置 | IP 冲突检测 |

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：网络工具箱                                       │
├──────────────────────────────────────────────────────────────┤
│ ToolTabs：Ping / TCP / HTTP / DNS / Traceroute / SSL / Subnet │
├───────────────────────────────┬──────────────────────────────┤
│ 左：输入参数表单               │ 右：执行结果                  │
├───────────────────────────────┴──────────────────────────────┤
│ 底部：执行历史 / 保存为监控项 / 创建告警规则                   │
└──────────────────────────────────────────────────────────────┘
```

### 工具页关键设计

不要让工具页只是一次性执行。

每个工具结果都应该支持：

```text
立即执行
保存为监控项
创建告警规则
复制结果
导出结果
查看历史
```

例如：

用户执行一次 `TCP 443` 检查后，可以点击：

```text
保存为监控项
```

系统自动创建：

```text
设备：example.com
探测类型：TCP
端口：443
周期：60s
失败阈值：连续 3 次失败触发告警
```

这样工具箱就能自然演进成监控平台。


### UI 建议

## P06：告警中心页

### 页面目标

告警中心不是单纯展示告警，而是帮助值班人员完成处置。

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：告警中心 / 当前告警 / 历史告警 / 规则 / 通知策略    │
├──────────────────────────────────────────────────────────────┤
│ AlertKpiRow：Critical / Major / Warning / 已确认 / 未确认      │
├──────────────────────────────────────────────────────────────┤
│ FilterBar：级别 / 状态 / 设备 / 类型 / 时间 / 关键字            │
├──────────────────────────────────────────────────────────────┤
│ AlertTable                                                    │
└──────────────────────────────────────────────────────────────┘
```

### 告警字段

| 字段 | 说明 |
|---|---|
| 级别 | Critical、Major、Warning、Info |
| 状态 | 未确认、已确认、处理中、已恢复、已关闭、已静默 |
| 告警名称 | 例如 Ping Down、High Latency、TCP Port Down |
| 对象 | 设备、IP、URL、端口 |
| 首次发生 | 首次触发时间 |
| 最近发生 | 最近一次触发时间 |
| 持续时间 | 持续多久 |
| 重复次数 | 告警重复次数 |
| 负责人 | 当前处理人 |
| 操作 | 确认、静默、关闭、查看详情 |

### 告警状态机

```text
触发
  -> 未确认
    -> 已确认
      -> 处理中
        -> 已恢复
          -> 已关闭

触发
  -> 静默中
    -> 静默到期
      -> 重新评估
```

### 告警规则示例

```text
Ping 连续失败 3 次 -> Critical
Ping 丢包率 > 30% 持续 5 分钟 -> Major
平均延迟 > 200ms 持续 10 分钟 -> Warning
TCP 端口连续失败 3 次 -> Critical
HTTP 状态码 >= 500 持续 3 次 -> Major
SSL 证书 15 天内过期 -> Warning
```

## P07：日志 / 事件页

### 页面目标

第一阶段不需要做复杂日志平台，但必须有事件时间线。

### 推荐先做事件，而不是完整日志平台

第一阶段事件类型：

| 类型 | 示例 |
|---|---|
| probe_success | 探测成功 |
| probe_failed | 探测失败 |
| alert_triggered | 告警触发 |
| alert_acknowledged | 告警确认 |
| alert_recovered | 告警恢复 |
| alert_closed | 告警关闭 |
| device_created | 创建设备 |
| device_updated | 修改设备 |
| maintenance_started | 进入维护 |
| maintenance_ended | 结束维护 |

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：事件日志 / 时间范围 / 实时流 / 导出                │
├──────────────────────────────────────────────────────────────┤
│ FilterBar：设备 / 事件类型 / 告警级别 / 关键字                 │
├──────────────────────────────────────────────────────────────┤
│ EventTimeline / EventTable                                   │
└──────────────────────────────────────────────────────────────┘
```

### 事件字段

| 字段 | 说明 |
|---|---|
| timestamp | 事件时间 |
| event_type | 事件类型 |
| severity | 严重级别 |
| device | 关联设备 |
| target | IP、URL、端口 |
| message | 事件说明 |
| raw_data | 原始结果 |

# 13. 关键可行性判断

## 13.1 技术可行性

可行。

原因：

```text
Ping / TCP / HTTP / DNS / Traceroute 都是成熟能力
FastAPI 很适合做网络工具 API
React + ECharts 很适合做仪表盘
PostgreSQL 足以支撑第一阶段设备、告警、事件数据
Redis + Celery 足以支撑周期探测任务
Docker Compose 便于独立部署
```

## 13.2 产品可行性

可行，但必须控制范围。

第一阶段不要做：

```text
完整拓扑自动发现
完整 SNMP 指标体系
完整 CMDB
复杂多租户
完整日志平台
复杂自动化变更平台
```

第一阶段应该做：

```text
设备
探测
趋势
告警
事件
工具
```

这 6 个对象足够形成产品闭环。

## 13.3 竞争差异

如果只是做 Ping 工具，价值不大。

如果做成：

```text
一次性网络工具
  + 可保存为监控项
  + 周期探测
  + 趋势分析
  + 告警闭环
  + 设备上下文
```

就会比普通网络工具网站更有产品价值。

# 14. 最终建议

`ost-network-tools` 最适合的第一阶段定位是：

> 一个轻量级、可自部署、从网络诊断工具自然演进到设备监控和告警平台的运维工作台。

开发优先级应为：

```text
仓库工程化
  -> UI 静态原型
    -> 设备与监控模型
      -> 网络工具执行
        -> 周期探测
          -> 告警闭环
            -> 指标趋势
              -> 日志事件
                -> 拓扑和 SNMP
```

最小闭环必须是：

```text
添加设备
  -> 配置 Ping / TCP / HTTP 监控
    -> 周期探测
      -> 记录指标
        -> 触发告警
          -> 告警确认 / 静默 / 关闭
            -> 形成事件时间线
              -> 首页展示健康状态
```

只要先把这个闭环做出来，后续再接入 SNMP、gNMI、Prometheus、拓扑、报表、通知渠道，产品就能自然升级成一个真正有企业级感觉的网络运维平台。
