# 网络运维 / 设备监控 / 仪表盘 UI 与前端设计建议

## 1. 产品定位

本产品不要只做成“图表堆叠的监控看板”，而应定位为：

> 网络运维值班工作台 + 设备资产管理台 + 告警处置中心 + 排障分析入口

用户进入系统后，应该能快速完成四件事：

1. 第一眼判断全网是否健康。
2. 第二步知道影响范围在哪里。
3. 第三步进入设备、链路、日志、指标上下文定位原因。
4. 第四步完成确认、派单、静默、维护、关闭和复盘。

所谓“企业级感觉”，不是单纯依靠深色主题、大屏动画或复杂图表，而是来自以下几个方面：

| 维度 | 企业级体现 |
|---|---|
| 信息层级 | 首页不堆数据，先健康态势，再异常，再下钻 |
| 状态一致性 | 全系统统一绿色、黄色、橙色、红色、灰色状态语义 |
| 可操作性 | 告警、设备、链路、日志都能进入详情、关联、处理 |
| 可追溯性 | 所有确认、静默、维护、关闭、变更操作都有审计记录 |
| 高密度但不混乱 | 表格、图表、标签、抽屉、筛选器布局稳定 |
| 权限与租户 | 支持角色、站点、区域、业务线隔离 |
| 性能可信 | 大表格分页/虚拟滚动，实时数据可暂停，图表可降采样 |
| 业务闭环 | 从发现问题到处置、工单、维护窗口、复盘形成闭环 |

## 3. 总体设计原则

### 3.1 不要做成“图表墙”

网络运维系统最常见的问题是首页塞满几十张图，导致用户不知道先看哪里。

首页应该先回答：

- 全网是否正常？
- 是否有严重故障？
- 故障影响哪个站点、业务、设备或链路？
- 当前谁需要处理？
- 能否一键进入排障上下文？

### 3.2 页面要围绕运维流程设计

推荐的用户路径：

```text
首页发现异常
  -> 点击严重告警
    -> 进入告警详情
      -> 查看影响范围
        -> 查看相关设备 / 链路 / 指标 / 日志
          -> 确认 / 分派 / 静默 / 维护 / 关闭
            -> 写入审计和复盘记录
```

### 3.3 统一状态语义

全系统必须保持状态色一致：

| 状态 | 颜色语义 | 文案 |
|---|---|---|
| normal | 绿色 | 正常 |
| warning | 黄色 | 警告 |
| major | 橙色 | 严重 |
| critical | 红色 | 故障 |
| maintenance | 蓝色 | 维护中 |
| unknown | 灰色 | 未知 |
| disabled | 灰色弱化 | 已禁用 |

注意：

- 状态不能只靠颜色表达，必须配合文字、图标或形状。
- 表格中使用小徽标，详情页中使用大状态徽标。
- 同一个状态在首页、设备页、链路页、告警页、日志页、资产页必须一致。

### 3.4 统一时间范围

所有页面都应支持统一时间范围：

- 最近 15 分钟
- 最近 1 小时
- 最近 6 小时
- 最近 24 小时
- 最近 7 天
- 自定义时间范围

从首页进入设备、链路、日志、告警详情时，时间范围应自动继承。

### 3.5 全局上下文要可传递

例如用户从设备详情页进入日志页时，日志页应自动带入：

```text
device_id
hostname
management_ip
site
interface
alert_id
time_range
```

这样用户不会在每个页面重复筛选。

## 4. 全局 UI 架构

### 4.1 页面外壳 AppShell

建议所有页面统一使用以下结构：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ GlobalTopBar：全局搜索 / 时间范围 / 自动刷新 / 站点切换 / 通知 / 用户 │
├───────────────┬──────────────────────────────────────┬───────────────┤
│ SideNav       │ MainContent                          │ ContextPanel  │
│ 一级菜单       │ 当前页面主体                           │ 右侧上下文面板   │
│ 二级菜单       │ 卡片 / 图表 / 表格 / 拓扑 / 日志         │ 详情 / Runbook  │
└───────────────┴──────────────────────────────────────┴───────────────┘
```

### 4.2 左侧导航建议

```text
驾驶舱
  - 全网总览
  - 站点总览
  - 大屏模式

监控
  - 设备监控
  - 接口与链路
  - 拓扑视图
  - 指标查询

告警
  - 当前告警
  - 告警规则
  - 通知策略
  - 维护窗口

资产
  - 设备资产
  - IPAM
  - 机柜 / 位置
  - 厂商 / 型号
  - 生命周期

日志与排障
  - 日志检索
  - 事件时间线
  - 关联分析

自动化
  - 作业中心
  - 配置备份
  - 配置对比
  - 变更任务

报表
  - SLA 报表
  - 可用率报表
  - 告警统计
  - 容量趋势

系统
  - 用户与角色
  - 数据源
  - 采集器
  - 审计日志
  - 系统设置
```

### 4.3 全局顶部栏组件

| 组件 | 位置 | 设计要求 |
|---|---|---|
| GlobalSearch | 顶部居中或左侧 | 支持搜索设备名、IP、接口、告警 ID、站点、标签 |
| TimeRangeSelector | 顶部右侧 | 支持 15m、1h、6h、24h、7d、自定义 |
| AutoRefreshToggle | 顶部右侧 | 支持 10s、30s、1m、5m、暂停刷新 |
| SiteSwitcher | 顶部右侧 | 支持总部、分支、IDC、租户、业务线切换 |
| NotificationBell | 顶部右侧 | 显示未确认告警数，点击打开告警抽屉 |
| UserMenu | 顶部右侧 | 个人设置、主题、语言、退出 |
| EnvironmentBadge | 顶部左侧 | 明确显示生产、测试、演示环境，避免误操作 |

## P01：首页 / 全网驾驶舱

### 页面目标

首页只解决一个问题：

> 当前网络是否健康，哪里最需要处理？

首页不要变成所有图表的堆叠页。首页应优先服务值班人员，而不是展示所有监控能力。

### 页面布局

```text
┌────────────────────────────────────────────────────────────────┐
│ 页面标题：全网驾驶舱       时间范围 / 自动刷新 / 大屏模式 / 导出 │
├────────────────────────────────────────────────────────────────┤
│ KPI Row：健康设备 / 故障设备 / 当前告警 / 业务可用率 / 平均延迟 / 链路在线 │
├───────────────────────────────┬────────────────────────────────┤
│ 全网健康分布 / 站点地图          │ 当前严重告警队列                 │
├───────────────────────────────┴────────────────────────────────┤
│ 趋势图区域：流量趋势 / 延迟丢包 / CPU 内存 / 告警趋势             │
├───────────────────────────────┬────────────────────────────────┤
│ 慢设备排行 / 异常链路排行         │ 近期恢复 / 维护窗口 / 采集器状态   │
└───────────────────────────────┴────────────────────────────────┘
```

### 组件清单

| 组件 | UI 编排 | 字段 / 内容 | 交互 |
|---|---|---|---|
| PageHeader | 页面顶部 | 标题、更新时间、数据源状态 | 刷新、导出、大屏模式 |
| KpiGrid | 第一行，4 到 6 张卡片 | 健康设备、故障设备、当前告警、业务可用率、平均延迟、在线链路 | 点击下钻到对应列表 |
| HealthCard | KPI 卡 | 主数字、环比、状态、微趋势 | Tooltip 解释口径 |
| SiteHealthMap | 左中 | 区域、站点、状态、告警数量 | 点击站点进入站点详情 |
| CriticalAlertQueue | 右中 | 严重级别、对象、持续时间、影响范围 | 确认、静默、进入详情 |
| TrafficTrendCard | 中下 | 入/出流量、峰值、95 分位 | 切换 1h / 24h / 7d |
| QualityTrendCard | 中下 | 延迟、抖动、丢包 | 阈值线、异常点标注 |
| ResourceTrendCard | 中下 | CPU、内存、磁盘、温度 | 支持设备类型筛选 |
| SlowNodesCard | 底部 | 慢响应设备排行 | 点击进入设备详情 |
| MaintenanceWindowCard | 底部 | 当前维护、即将开始维护 | 查看维护影响范围 |
| CollectorStatusCard | 底部 | 采集器在线数、延迟、失败率 | 进入采集器管理 |

### KPI 卡建议

顶部建议放 4 到 6 张 KPI 卡，不要太多。

| 卡片 | 主指标 | 辅助信息 | 点击后跳转 |
|---|---|---|---|
| 健康设备数 | 正常设备数量 | 占比、较 1 小时前变化 | 设备列表，状态=正常 |
| 故障设备数 | 故障设备数量 | 严重级别、站点分布 | 设备列表，状态=故障 |
| 当前告警数 | 未恢复告警数量 | Critical / Major 数量 | 告警中心 |
| 业务可用率 | 百分比 | SLA 达标状态 | SLA 页面 |
| 平均延迟 | ms | 环比变化 | 链路质量页面 |
| 在线链路数 | 在线链路数量 | 异常链路数量 | 接口与链路页面 |

### 首页视觉建议

- KPI 卡片最多 6 张，不超过一屏宽度。
- 主数字使用 28px 到 36px。
- 说明文字使用 12px 到 13px。
- 卡片之间留白统一，避免“报表墙”感觉。
- 首页默认展示最近 24 小时趋势。
- 首页适合参考：
  - Uptime Kuma 的状态表达。
  - Netdata 的实时指标密度。
  - Grafana 的时间范围和面板结构。

### 首页验收标准

- 5 秒内能判断是否有严重故障。
- 所有红色状态都能点击下钻。
- 首页无权限用户不会看到越权站点数据。
- 自动刷新不会打断用户正在查看的 Tooltip、抽屉、筛选条件。
- 任一图表数据为空时有明确空状态，而不是空白卡片。

## P02：站点总览页

### 页面目标

按区域、站点、机房、业务线展示健康状态，帮助用户判断：

> 问题是否集中在某个地域或站点？

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：站点总览 / 区域筛选 / 业务筛选 / 时间范围          │
├──────────────────────────────────────────────────────────────┤
│ 区域健康 KPI：正常站点 / 异常站点 / 维护中 / 采集异常          │
├───────────────────────┬──────────────────────────────────────┤
│ 左：区域树 / 站点列表    │ 右：地图 / 站点卡片矩阵               │
├───────────────────────┴──────────────────────────────────────┤
│ 站点健康明细表：站点、设备数、告警数、链路数、可用率、负责人     │
└──────────────────────────────────────────────────────────────┘
```

### 组件清单

| 组件 | UI 编排 | 设计要求 |
|---|---|---|
| RegionTree | 左侧 | 按国家 / 城市 / 园区 / IDC 层级展示 |
| SiteCardMatrix | 主区域 | 每个站点一张状态卡，展示健康度、告警数、链路状态 |
| SiteMap | 主区域 | 站点地理分布，颜色表示状态 |
| SiteHealthTable | 底部 | 支持排序、筛选、导出 |
| SiteDetailDrawer | 右侧抽屉 | 点击站点后显示设备、链路、告警、负责人 |

### 站点卡字段

| 字段 | 说明 |
|---|---|
| 站点名称 | 例如：华东 IDC、北京总部、深圳分支 |
| 状态 | 正常、警告、故障、维护中 |
| 设备数量 | 总设备 / 异常设备 |
| 当前告警 | Critical / Major / Warning 数 |
| 链路状态 | 总链路 / 异常链路 |
| 可用率 | 最近 24 小时或 7 天 |
| 负责人 | 站点负责人或值班团队 |

### 参考项目

- 站点和资产模型参考 NetBox / Nautobot。
- 传统 NMS 的站点/设备监控思路参考 LibreNMS、OpenNMS。
- 地图和统计卡片可参考 Grafana / Superset 的 Dashboard 组织方式。

## P03：设备监控列表页

### 页面目标

用于快速定位设备状态、筛选异常设备、进入设备详情。

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：设备监控 / 新增关注 / 导出 / 批量操作              │
├──────────────────────────────────────────────────────────────┤
│ FilterBar：站点 / 设备类型 / 厂商 / 状态 / 标签 / 负责人 / 搜索 │
├──────────────────────────────────────────────────────────────┤
│ DeviceTable：设备列表                                         │
├──────────────────────────────────────────────────────────────┤
│ Pagination / SavedViews / ColumnSettings                     │
└──────────────────────────────────────────────────────────────┘
```

### 表格字段建议

| 字段 | 类型 | 说明 |
|---|---|---|
| 状态 | StatusBadge | 正常、警告、严重、故障、维护中、未知 |
| 设备名 | Link | 点击进入设备详情 |
| 管理 IP | MonoText | 使用等宽字体 |
| 站点 | Tag | 可点击筛选 |
| 角色 | Tag | 核心、汇聚、接入、防火墙、AP、服务器 |
| 厂商 / 型号 | Text | 支持排序 |
| OS 版本 | Text | 可用于风险排查 |
| CPU / 内存 | MiniBar | 小型进度条 |
| 接口异常数 | NumberBadge | 点击进入接口列表 |
| 当前告警 | Badge | 点击进入告警中心并带过滤条件 |
| 最后采集时间 | RelativeTime | 超时显示黄色或红色 |
| 操作 | ActionMenu | 查看、静默、维护、加入关注、打开终端入口 |

### 筛选器建议

| 筛选项 | 说明 |
|---|---|
| 站点 | 支持区域、IDC、分支机构 |
| 设备类型 | 路由器、交换机、防火墙、AP、服务器 |
| 厂商 | Cisco、Huawei、H3C、Juniper、Fortinet 等 |
| 状态 | 正常、警告、故障、维护中、未知 |
| 标签 | 生产、办公、核心、边缘、专线 |
| 负责人 | 团队或个人 |
| 采集状态 | 正常、延迟、失败、未采集 |
| 关键字 | 设备名、IP、序列号、接口描述 |

### 前端实现建议

- 表格必须服务端分页、服务端排序、服务端筛选。
- 支持列显示/隐藏、列顺序调整、固定列。
- 支持保存视图，例如：
  - 核心设备异常
  - 华东 IDC 防火墙
  - 近 1 小时未采集设备
- 大规模数据场景使用虚拟滚动。
- 表格能力可参考 TanStack Table。
- 后台骨架可参考 Ant Design Pro 或 react-admin。

## P04：设备详情页

### 页面目标

设备详情页是排障核心页面，应把以下信息放在同一个上下文里：

- 设备基础信息
- 实时指标
- 接口状态
- 当前告警
- 相关日志
- 配置备份
- 资产关系
- 变更历史

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ DeviceHeader：设备名 / 状态 / IP / 站点 / 角色 / 负责人 / 操作 │
├──────────────────────────────────────────────────────────────┤
│ SummaryCards：可用率 / 当前告警 / CPU / 内存 / 接口异常 / Uptime│
├──────────────────────────────────────────────────────────────┤
│ Tabs：概览 / 接口 / 指标 / 告警 / 日志 / 配置 / 关系 / 变更历史 │
├──────────────────────────────────────────┬───────────────────┤
│ TabContent                                │ RightContextPanel │
│ 图表 / 表格 / 时间线 / 拓扑                 │ 负责人 / Runbook   │
└──────────────────────────────────────────┴───────────────────┘
```

### 设备头部字段

| 字段 | 说明 |
|---|---|
| 设备名称 | hostname |
| 状态 | 正常、警告、故障、维护中 |
| 管理 IP | 可复制 |
| 站点 | 可点击进入站点页 |
| 设备角色 | 核心、汇聚、接入、防火墙、AP |
| 厂商型号 | 厂商、型号、序列号 |
| OS 版本 | 系统版本 |
| 负责人 | 团队或个人 |
| 最后采集时间 | 相对时间 + 绝对时间 Tooltip |

### Tab 设计

| Tab | 内容 | 组件 |
|---|---|---|
| 概览 | 基础信息、健康状态、最近异常、关键指标 | InfoCard、MetricCard、AlertMiniList |
| 接口 | 接口列表、速率、错误包、丢包、上下联关系 | InterfaceTable、TrafficSparkline |
| 指标 | CPU、内存、磁盘、温度、电源、风扇 | MetricChartGrid |
| 告警 | 当前告警、历史告警、确认记录 | AlertTable、AlertTimeline |
| 日志 | 设备日志、系统日志、采集日志 | LogQueryPanel、LogList |
| 配置 | 当前配置、备份配置、配置差异 | ConfigViewer、DiffViewer |
| 关系 | 上联、下联、所在机柜、所属业务 | RelationGraph、TopologyMiniMap |
| 变更历史 | 字段变更、维护记录、自动化任务 | AuditTimeline |

### 关键交互

- 设备 Header 中状态必须始终可见。
- 任何指标异常点都能跳转到同时间范围的日志页。
- 点击接口名称进入接口详情抽屉。
- 点击告警进入告警详情页，保留设备上下文。
- 进入配置页时默认只读。
- 修改配置必须走变更流程。
- 右侧 ContextPanel 显示负责人、维护窗口、关联文档、Runbook。

### 参考项目

- 详情结构参考 NetBox。
- 指标图表参考 Netdata / Grafana。
- 日志、指标、告警上下文联动参考 SigNoz。
- 自动化和变更扩展参考 Nautobot。

### 组件清单

| 组件 | 设计要求 |
|---|---|

## P06：网络拓扑页

### 页面目标

拓扑页不是装饰图，而是用于回答：

- 哪些设备互联？
- 哪条链路异常？
- 当前告警影响了哪些上下游？
- 某个站点或业务的路径在哪里中断？

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ TopologyToolbar：层级 / 布局 / 筛选 / 搜索 / 保存视图 / 全屏    │
├───────────────┬────────────────────────────────┬─────────────┤
│ 左侧过滤器      │ 中央拓扑画布                     │ 右侧详情抽屉  │
│ 站点/业务/状态   │ 节点 / 边 / 分组 / 状态 / 告警     │ 设备/链路详情 │
├───────────────┴────────────────────────────────┴─────────────┤
│ 底部时间轴：拓扑变化 / 告警发生 / 链路恢复 / 配置变更            │
└──────────────────────────────────────────────────────────────┘
```

### 拓扑节点设计

| 元素 | 设计要求 |
|---|---|
| 设备节点 | 图标区分路由器、交换机、防火墙、AP、服务器 |
| 节点状态 | 外圈颜色表示状态，不只依赖填充色 |
| 链路边 | 线宽表示带宽，颜色表示状态，动画表示实时流量 |
| 分组 | 支持按站点、机房、业务、VRF 分组 |
| 告警标记 | 节点右上角显示告警数量 |
| 悬浮卡片 | 展示 IP、型号、角色、接口、当前告警 |
| 右侧详情 | 点击节点或链路后打开详情抽屉 |

### 交互规则

- 默认展示站点级拓扑，不要一次渲染全网所有节点。
- 支持逐层展开：区域 -> 站点 -> 机房 -> 设备 -> 接口。
- 支持“只看异常路径”。
- 支持“路径查询”：源设备到目的设备。
- 支持“时间回放”：查看某个故障发生前后的拓扑变化。
- 支持保存拓扑视图，供不同团队使用。

### 参考项目

- 关系图能力参考 AntV G6。
- 可拖拽节点式 UI 参考 React Flow / xyflow。
- 网络对象关系参考 NetBox / Nautobot。
- 查询式路径分析参考 SuzieQ。

## P07：告警中心页

### 页面目标

告警中心不是“告警列表”，而是：

> 值班处置工作台

用户需要在这里完成：

1. 看当前最严重的问题。
2. 判断是否重复告警或告警风暴。
3. 确认告警。
4. 静默或维护。
5. 分派负责人。
6. 进入详情定位。
7. 关闭并记录原因。

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：告警中心 / 当前告警 / 历史告警 / 规则 / 通知策略    │
├──────────────────────────────────────────────────────────────┤
│ AlertKpiRow：Critical / Major / Minor / Warning / 已确认 / 未确认│
├──────────────────────────────────────────────────────────────┤
│ FilterBar：级别 / 状态 / 站点 / 设备 / 业务 / 时间 / 标签 / 搜索 │
├──────────────────────────────────────────────────────────────┤
│ AlertGroupTabs：按告警分组 / 按设备分组 / 按站点分组 / 按规则分组 │
├──────────────────────────────────────────────────────────────┤
│ AlertTable：告警列表                                          │
└──────────────────────────────────────────────────────────────┘
```

### 告警表格字段

| 字段 | 说明 |
|---|---|
| 级别 | Critical、Major、Minor、Warning、Info |
| 状态 | 未确认、已确认、处理中、已静默、已恢复、已关闭 |
| 告警名称 | 例如 Interface Down、High CPU、Packet Loss |
| 对象 | 设备、接口、链路、服务 |
| 站点 / 业务 | 显示影响范围 |
| 首次发生 | 首次触发时间 |
| 最近发生 | 最近一次触发时间 |
| 持续时间 | 例如 37m |
| 重复次数 | 告警风暴判断 |
| 负责人 | 值班人或团队 |
| 操作 | 确认、分派、静默、详情、关闭 |

### 告警状态机建议

```text
触发
  -> 未确认
    -> 已确认
      -> 处理中
        -> 已恢复
          -> 已关闭

触发
  -> 已静默
    -> 静默到期
      -> 未确认

触发
  -> 维护中
    -> 维护结束
      -> 根据当前状态重新评估
```

### 关键交互

- 告警默认按严重级别和影响范围排序，不按时间简单倒序。
- 支持告警聚合，避免 1000 条接口告警淹没主问题。
- 支持批量确认、批量静默。
- 静默必须设置原因和过期时间。
- 关闭告警必须选择关闭原因。
- 已恢复但未关闭的告警要单独分区，避免遗漏复盘。
- 告警风暴时显示全局 Banner，而不是只增加列表数量。

### 参考项目

- 简洁状态与通知参考 Uptime Kuma。
- 企业级问题列表、触发器、模板参考 Zabbix。
- 网络告警与性能整合参考 OpenNMS。
- 告警规则和仪表盘联动参考 Grafana。

## P11：自动化 / 作业中心页

### 页面目标

将网络运维从“看问题”推进到“可控地执行动作”。

### 页面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ PageHeader：作业中心 / 新建作业 / 模板 / 审批 / 执行历史        │
├──────────────────────────────────────────────────────────────┤
│ JobKpiRow：运行中 / 成功 / 失败 / 待审批 / 今日执行              │

│ JobTable：作业名称 / 类型 / 目标 / 状态 / 发起人 / 开始时间 / 结果│
├──────────────────────────────────────────────────────────────┤
│ JobDetailDrawer：参数 / 日志 / 输出 / 影响对象 / 回滚方案        │
└──────────────────────────────────────────────────────────────┘
```

### 作业类型

| 类型 | 示例 |
|---|---|
| 配置备份 | 拉取设备配置 |
| 配置对比 | 当前配置 vs 上次配置 |
| 连通性检查 | Ping、traceroute、BGP 邻居 |
| 资产同步 | 从 NetBox / Nautobot / CMDB 同步 |

- 批量变更必须支持审批。
- 执行前必须展示影响范围。
- 执行中要有实时日志。
- 失败要有失败原因和重试入口。
- 高风险任务必须要求回滚方案。

### 参考项目

- 自动化平台能力参考 Nautobot。
- 资产数据源参考 NetBox。

```text
│ ReportTemplateCards：SLA / 可用率 / 告警统计 / 容量趋势 / 巡检  │
├──────────────────────────────────────────────────────────────┤
│ ReportBuilder：时间范围 / 站点 / 业务 / 指标 / 分组 / 图表类型   │
├──────────────────────────────────────────────────────────────┤
│ ReportPreview：图表 + 表格 + 摘要                              │
└──────────────────────────────────────────────────────────────┘
```

### 报表类型

| 报表 | 内容 |
|---|---|

| 容量趋势 | 带宽利用率、CPU、内存、磁盘增长趋势 |

### 参考项目

- 仪表盘与图表组合参考 Grafana。
- 报表构建和数据探索可参考 Apache Superset。
- 表格导出和管理后台结构参考 Ant Design Pro / react-admin。

## 6.1 KPI 卡片

### 结构

```text
┌────────────────────┐
│ 标题          状态点 │
│ 12,438             │
│ ↑ 3.2% vs 1h ago   │
```

### 字段

| 字段 | 说明 |
|---|---|
| title | 指标名称 |
| value | 主数值 |
| unit | 单位 |
| trend | 环比或同比 |
| status | normal / warning / critical / unknown |
| sparkline | 小趋势图 |
| tooltip | 指标口径说明 |
| drilldown | 点击跳转目标 |

### 设计要求

- 主数字必须明显。
- 不要每张卡使用不同颜色，颜色只表达状态。
- 有异常时显示原因提示，例如“因 3 台核心交换机不可达导致可用率下降”。
- 卡片点击必须能下钻，不做纯展示卡片。

## 6.2 图表卡片

### 图表卡片结构

```text
┌──────────────────────────────────────┐
│ 标题 / 当前值 / 阈值 / 操作菜单        │
├──────────────────────────────────────┤

│ 图例 / 时间范围 / 异常点 / 数据来源     │
```

### 设计要求

- 图表默认显示阈值线。
- 异常点必须有标注。
- 图例不能过多，超过 5 条曲线要分组。
- 支持 hover 查看精确值。
- 支持局部放大。
- 支持与日志、告警联动跳转。
- 数据量大时后端降采样，前端不直接渲染海量点。

## 6.3 状态徽标 StatusBadge

### 设计要求

- 状态不能只靠颜色，必须配合文字或图标。
- 表格里用小徽标，详情页用大徽标。
- 同一状态在所有页面必须一致。
- “维护中”和“已确认”不要用红色，避免误导。
- “未知”不要等同于“正常”，应使用灰色并提示采集状态。

## 8.4 实时数据处理

- 告警数量、设备状态、采集器状态适合 WebSocket / SSE。

- 用户打开 Tooltip、抽屉、表格选择时，自动刷新不能打断操作。
- 所有实时页面要有“暂停刷新”按钮。
- 网络断开时顶部显示数据连接状态。

## 8.5 性能要求

| 场景 | 要求 |
|---|---|

| 日志列表 | 虚拟滚动，不一次性渲染大量日志 |
| 拓扑 | 默认按站点聚合，避免全网全量渲染 |
| 图表 | 后端降采样，前端最多渲染合理点数 |
| 筛选 | 防抖查询 |
| 大屏 | 独立路由，隐藏复杂操作入口 |

# 11. 推荐落地组合

最推荐的组合参考是：

```text
首页健康总览：Uptime Kuma + Grafana

```

这套设计的核心不是“像某个开源项目”，而是吸收它们成熟的产品结构：

- Uptime Kuma 的状态直观性。
- Netdata 的实时指标密度。
- Grafana 的仪表盘体系。
- NetBox / Nautobot 的网络对象建模。
- SigNoz 的排障上下文。
- SuzieQ 的网络问题查询思路。
- Zabbix / OpenNMS 的告警处置流程。
- Ant Design Pro / react-admin 的企业后台结构。
- TanStack Table 的复杂表格能力。
- ECharts / G6 / React Flow 的图表和拓扑表达能力。

最终目标是做出一个真正像企业级运维平台的产品，而不是普通监控模板。

# 12. 最终建议

开发团队不要先追求“炫酷大屏”，而应先做出稳定、清晰、可操作的企业级后台。

优先级应是：

```text
信息架构
  > 运维流程
    > 组件一致性
      > 数据联动
        > 视觉美化
          > 大屏效果
```

产品第一阶段一定要做到：

1. 首页能看出全网是否健康。
2. 设备列表能快速筛出异常设备。
3. 设备详情能看到指标、接口、告警、日志。
4. 告警中心能完成确认、静默、分派、关闭。
5. 日志页能从设备和告警上下文跳转过来。
6. 资产数据能支撑监控、拓扑、权限和自动化。
7. 所有敏感操作都有审计。
8. 所有状态、时间、筛选、表格、抽屉交互保持一致。

这样做出来的系统，才会有真正的“企业级感觉”。
